## Task Goal

Improve the Clojure/JVM + Java2D + native FBInk proof of concept toward fast warm rendering suitable for an e-reader:

- target sub-1-second page renders after JVM warmup;
- keep a long-lived JVM process acceptable, because cold startup is very slow;
- add Clojure code reloading so updated code can be rsynced to the Kobo and reloaded without restarting the JVM process;
- document status and findings in the repo, originally noted as `clojure-eink/STATUS.md`.

## Immediate Implementation Tasks

Add benchmark modes to `src/clj/ol/project.clj`:

- `--renders N` or `--repeat N`;
- render the same page multiple times inside one JVM;
- log timings for each iteration separately;
- separate, if possible:
  - image allocation;
  - background fill;
  - text layout;
  - glyph draw;
  - image-to-gray8 conversion;
  - native present;
- allow `--no-present` for render-only benchmarking;
- allow presenting only the final iteration with `--present-last`.

Then:

1. Package `target/dist/` again after edits.
2. Rsync to Kobo:

   ```sh
   rsync -rtv --delete --no-owner --no-group --no-perms target/dist/ \
     root@kobo-lan:/mnt/onboard/clojure-eink-demo/
   ```

3. Run benchmarks over SSH/tmux:

   ```sh
   ssh root@kobo-lan
   cd /mnt/onboard/clojure-eink-demo
   time ./run-demo.sh --renders 5 --no-present
   time ./run-demo.sh --renders 5 --present-last --no-wait --no-flash
   time ./run-demo.sh --renders 5 --present-last
   ```

4. Analyze whether Java2D warm render improves after the first render.

If warm renders remain slow, propose or implement next directions:

- cache/reuse `BufferedImage`;
- cache fonts/layout;
- avoid `LineBreakMeasurer` for a simple benchmark;
- compare Java2D text drawing vs simple fill/rectangle drawing;
- consider a native FreeType/HarfBuzz path if Java2D remains too slow.
